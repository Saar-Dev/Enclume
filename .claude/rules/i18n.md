# Localisation (i18n)

Lire `docs/SYSTEME/LOCALISATION.md` pour le détail. Pendant le chantier de résorption de la dette
actuelle, lire aussi `docs/PLAN_LOCALISATION.md`.

## Invariants

- Aucun texte visible par l'utilisateur codé en dur dans un `.jsx` — enfant JSX littéral,
  `placeholder=`, `title=`, `aria-label=`, `alt=`, `<option>`, message d'erreur, tooltip. Toujours
  `useTranslation()` → `t('section.cle')`.
- Ajouter la clé dans le namespace concerné (`client/src/locales/*.json`) **avant** de l'utiliser dans
  le JSX, jamais l'inverse. Vérifier d'abord qu'elle n'existe pas déjà (Règle 2, une information = un
  seul endroit).
- `fr.json` seul est réservé au namespace transverse (`common`). Un domaine dense a son propre fichier
  namespace (`charSheet.json`, `combat.json`, `builder.json`, `creation.json`) — jamais tout entassé
  dans un fichier unique qui grossit sans fin.
- Le serveur n'émet jamais de texte FR figé destiné à l'utilisateur. Un message système passe
  `system: true` + `i18nKey`, résolu côté client via `t()` (pattern `socketCombatHelpers.js` /
  `useSessionSocket.js`).
- L'anglais est gelé : `en.json` n'est pas chargé (`supportedLngs: ['fr']`) et n'a plus d'obligation de
  synchronisation avec le FR. Ne pas bloquer une tâche pour lui, ne pas le supprimer.
